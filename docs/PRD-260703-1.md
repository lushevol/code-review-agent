## 1. Extracted requirement

### Initiative

**PR Guardian Copilot: Centralized AI Code Review for Bug, Security, and Compliance Risk**

### Business description

> A centralized AI copilot reviews pull requests on submission and provides prioritized feedback on bugs, security issues, compliance risks, and CVEs. It posts review comments, creates remediation tasks for high-risk findings, and supports policy-based merge blocking until critical tasks are resolved. It also provides a dashboard for PR quality, risk, and productivity tracking.

### Expected deliverables

1. Enabled for **2–3 pilot repositories**.
2. Automated PR comments for risk findings, including **severity levels**.
3. Automatically created tracking tasks for **Critical and High** findings.
4. PR merge is blocked while **blocking-risk remediation tasks remain open**.
5. A live dashboard showing:

   * PR open-to-close cycle time
   * First-review response time
   * Findings by severity
   * Open versus closed remediation tasks
   * CVE trend
   * Top PR commenters
   * Top contributors

### Expected benefits

* Faster review turnaround
* Consistent risk controls across teams
* Earlier CVE and security detection
* Improved auditability
* Measurable engineering productivity improvement
* Transparent PR analytics

# 2. Requirement analysis

## A. Core functional requirements

| Capability             | Interpreted requirement                                                                                                             |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| PR event detection     | The agent must automatically start when a PR is created or updated.                                                                 |
| Code analysis          | Analyze the PR diff for bugs, security vulnerabilities, compliance violations and dependency risks.                                 |
| Context-aware analysis | Review findings against repository code, configuration, dependencies and internal engineering standards—not only the changed lines. |
| Risk classification    | Every finding must have a category, severity, confidence level and remediation recommendation.                                      |
| PR commenting          | Publish findings as PR-level summaries and, where possible, inline comments.                                                        |
| Task creation          | Create remediation work items for Critical and High findings automatically.                                                         |
| Merge governance       | Prevent completion of the PR while unresolved blocking findings or linked tasks remain open.                                        |
| Re-review              | Re-run analysis after new commits and close or supersede findings that have been remediated.                                        |
| Dashboard              | Aggregate PR, risk, remediation and productivity metrics across pilot repositories.                                                 |
| Audit trail            | Preserve the analysis result, model/rule version, evidence, decision and remediation status.                                        |

A suitable normalized finding should include:

```text
Finding
├── Finding ID
├── PR and repository
├── File and line range
├── Category
│   ├── Bug
│   ├── Security
│   ├── Compliance
│   └── CVE / dependency
├── Severity
│   ├── Critical
│   ├── High
│   ├── Medium
│   ├── Low
│   └── Informational
├── Confidence
├── Evidence
├── Business impact
├── Recommended remediation
├── Blocking status
├── Linked remediation task
├── Resolution status
└── Review engine and rule version
```

## B. Recommended review workflow

```text
PR Created or Updated
        │
        ▼
Eligibility and Build Gate
        │
        ▼
Collect PR Diff and Repository Context
        │
        ├── Static analysis results
        ├── Dependency and CVE results
        ├── Repository standards
        ├── Related source code
        └── Historical review context
        │
        ▼
Run Review Rules and AI Analysis
        │
        ▼
Deduplicate and Prioritize Findings
        │
        ▼
Publish PR Summary and Inline Comments
        │
        ├── Critical / High
        │       └── Create remediation task
        │
        └── Medium / Low
                └── Recommendation only
        │
        ▼
Evaluate Merge Policy
        │
        ├── Blocking issue remains → Block PR
        └── No blocking issue → Allow normal approval flow
        │
        ▼
New Commit Submitted
        │
        └── Re-review and reconcile findings
```

The requirement says “on submission,” but this should be interpreted as both initial submission and subsequent source-branch updates. Otherwise, a developer could push remediation code without the agent validating whether the issue was actually fixed.

# 3. Source of findings

The requirement should not imply that the LLM independently detects everything. A defensible design combines deterministic tools and AI reasoning.

| Finding type                     | Recommended source                                                                   |
| -------------------------------- | ------------------------------------------------------------------------------------ |
| CVEs and vulnerable dependencies | Software composition analysis, dependency scanner or approved vulnerability database |
| Secrets                          | Secret scanning tool                                                                 |
| Static code security issues      | SonarQube, SAST or approved security scanner                                         |
| Test coverage and quality gate   | SonarQube and pipeline results                                                       |
| Compliance rules                 | Version-controlled policy-as-code rules                                              |
| Logic bugs and contextual risks  | AI review using PR diff and repository context                                       |
| Architectural violations         | AI plus repository-specific architecture rules                                       |
| Prioritization and explanation   | AI correlation and summarization layer                                               |

The AI copilot should correlate, explain and prioritize scanner results rather than replace authoritative security scanners. This improves accuracy and auditability.

# 4. Merge-blocking requirements

The image states that the PR should be blocked when blocking-risk tasks remain open. That requires a precise policy.

A recommended default is:

| Severity      |            Task creation | Default merge policy             |
| ------------- | -----------------------: | -------------------------------- |
| Critical      |                Automatic | Block                            |
| High          |                Automatic | Block, unless approved exception |
| Medium        | Optional or configurable | Warning                          |
| Low           |        No automatic task | Advisory                         |
| Informational |                       No | Advisory                         |

The branch policy should check the current review state rather than only whether a task exists.

For example:

```text
Merge allowed when:

- Latest PR commit has been reviewed
- No unresolved Critical finding exists
- No unresolved blocking High finding exists
- All required remediation tasks are Closed, Resolved or formally Waived
- Required pipeline and quality gates have passed
```

A formal override path is needed for false positives, accepted risks and urgent releases. An override should require:

* Authorized approver
* Business justification
* Expiry or review date
* Finding status set to `Risk Accepted` or `False Positive`
* Full audit history

Without this exception process, teams may bypass the agent or treat it as an obstruction.

# 5. Dashboard requirements

The dashboard should separate **risk outcomes** from **developer productivity metrics**.

## Risk and quality metrics

* Findings by severity, repository and category
* Critical and High findings over time
* Open, resolved, waived and false-positive findings
* CVE trends by severity
* Mean time to remediate
* Reopened findings
* Findings per PR or per changed lines
* Blocking-rate trend
* False-positive rate
* Rule-level detection and acceptance rate

## PR delivery metrics

* PR open-to-close cycle time
* Time to first human review
* Time to first automated review
* Review-to-remediation time
* Number of review iterations
* PR throughput
* Percentage of PRs reviewed by the agent
* Percentage of PRs merged without blocking findings

## Adoption metrics

* Pilot repositories onboarded
* Active contributors
* Review comments accepted or addressed
* Top PR commenters
* Top contributors
* Feedback participation rate

“Top contributors” and “top commenters” should be handled carefully. They may be useful operational metrics, but using them as individual performance rankings can incentivize comment volume rather than review quality. Repository- and team-level views are generally safer.

# 6. Missing or ambiguous requirements

The supplied requirement gives a clear business direction, but the following decisions are still required.

### Review scope

* Is only the changed diff analyzed?
* Can the agent read the full repository?
* Should it inspect configuration, dependency manifests and tests?
* Are generated files and vendor code excluded?

For meaningful code review, repository-aware context should be included. Reviewing only the diff will produce “lack of context” findings and a higher false-positive rate.

### Supported technology

The pilot should specify languages and frameworks, likely:

* React and TypeScript
* Java and Spring
* Build and dependency files
* Infrastructure or deployment configuration where relevant

### CVE integration

“CVE detection” needs an authoritative data source and scanner. The requirement should identify:

* Approved vulnerability database
* Scan frequency
* Dependency reachability support
* Treatment of transitive dependencies
* Accepted-risk and suppression lifecycle

### Policy ownership

It must be clear who owns:

* Security rules
* Compliance rules
* Engineering-quality rules
* Severity definitions
* Blocking policy
* Exceptions and waivers

### Task lifecycle

The requirement should define:

* Work-item type
* Assignment strategy
* Due date or SLA
* Linkage to the PR and finding
* Statuses considered resolved
* Whether task closure automatically re-evaluates the PR
* Behaviour when the PR is abandoned

### Review latency

A review SLA should be established. For example:

* Initial acknowledgement within 30 seconds
* Standard PR review completed within 3–5 minutes
* Large PRs may enter an asynchronous review state
* Failure must not silently appear as a successful review

### False-positive handling

Each finding should support feedback such as:

* True positive
* False positive
* Lack of context
* By design
* Risk accepted
* Already addressed

This is necessary for prompt and rule improvement and for measuring whether the copilot is genuinely useful.

### Data and model governance

Because this operates on bank source code, requirements should cover:

* Approved LLM endpoint
* No code retention outside the controlled environment
* Prompt and response logging rules
* Sensitive-code masking where necessary
* Model and prompt versioning
* Access control by repository
* Complete audit logging

# 7. Pilot scope and acceptance criteria

A practical pilot can be expressed as follows.

## Pilot scope

* Two to three repositories
* At least one React/TypeScript repository
* At least one Java/Spring repository
* Azure DevOps PR integration
* Existing pipeline and SonarQube integration
* Work-item creation
* Branch-policy status reporting
* Basic analytics dashboard

## Minimum acceptance criteria

| Area          | Acceptance criterion                                                              |
| ------------- | --------------------------------------------------------------------------------- |
| Triggering    | Every eligible PR and new commit triggers a review automatically.                 |
| Coverage      | At least 95% of eligible pilot PRs receive a completed review result.             |
| Response time | The first automated response is posted within the agreed SLA.                     |
| Findings      | Findings contain severity, confidence, evidence and remediation guidance.         |
| Comments      | Findings are posted without duplicate comments across repeated runs.              |
| Tasks         | Critical and High findings create traceable remediation work items.               |
| Blocking      | PR completion is blocked when the latest review has unresolved blocking findings. |
| Resolution    | A subsequent commit can resolve or supersede an earlier finding.                  |
| Override      | Authorized users can waive a finding with justification and audit history.        |
| Dashboard     | All required metrics are visible by repository and reporting period.              |
| Auditability  | Each result records the source commit, rules, model and review timestamp.         |
| Quality       | False-positive and developer-feedback metrics are collected during the pilot.     |

# 8. Recommended consolidated requirement

> PR Guardian Copilot shall centrally review eligible Azure DevOps pull requests when they are created and whenever their source commit changes. It shall combine deterministic engineering and security scanners with repository-aware AI analysis to identify bugs, security vulnerabilities, compliance violations and dependency risks.
>
> Each finding shall contain a category, severity, confidence level, supporting evidence and recommended remediation. The system shall publish prioritized PR summaries and inline comments, automatically create linked remediation tasks for Critical and High findings, and report a branch-policy status that blocks merging while unresolved blocking findings remain against the latest commit.
>
> The solution shall support authorized risk acceptance and false-positive workflows with a complete audit trail. It shall provide dashboard reporting for PR cycle time, first-review response time, findings by severity, remediation status, CVE trends, review activity and adoption metrics. The initial pilot shall cover two to three repositories and shall measure review coverage, latency, detection quality, false-positive rate, remediation time and developer adoption.

The most important additions to the original requirement are **reviewing every new commit, whole-repository context, deterministic CVE integration, false-positive handling, exception governance and auditable policy evaluation**. Without those elements, the solution may post comments, but it will not be reliable enough to serve as a genuine PR quality gate.
